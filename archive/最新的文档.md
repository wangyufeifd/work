## 1. 项目概述

### 1.1 项目定位

Harbour 是面向多市场、多交易所的**统一持仓 & 保证金计算中台**。平台统一对接各区域异构数据源，标准化数据格式、计算逻辑与接口规范，为下游交易、结算、风控、监控系统提供一致、可靠的持仓、保证金、盈亏风险数据。

### 1.2 解决的核心痛点

- 各交易所 / 市场数据源协议、字段、业务语义差异大，跨市场接入重复适配成本极高；
- 传统架构为每个市场独立开发全套代码，重复工作量大、维护成本高；
- 下游系统需要对接多套异构接口，接口迁移、联调、运维负担重；
- 原有架构无法支撑 GMI + 多级子账户统一管理、跨市场统一风控计算。

### 1.3 整体架构理念

> **统一数据模型 + 隔离适配器层 + 可复用计算模块 + Kafka 流驱动架构 + CQRS 查询服务 + 标准化扩展机制**

---

## 2. 顶层强制设计原则

平台所有组件、代码、流程必须严格遵守以下规则，**无例外**：

1. 所有上游异构数据，必须经由适配器统一转换为一套标准 Protobuf (Octane) 模型，所有计算模块仅消费该统一模型；
2. 严格按分层划分职责，数据接入、数据标准化、消息传输、业务计算、数据查询完全解耦；
3. 平台内部流式数据仅使用 Octane Protobuf 二进制格式作为唯一传输标准；
4. 所有市场差异化逻辑、私有语义，统一在适配器层屏蔽，通用计算模块不编写市场硬编码；
5. 过渡期允许两套数据模型并存，必须建立同步、校验、下线规则，杜绝技术债务与语义漂移；
6. 核心数据模型长期稳定不变，市场特有业务语义通过标准化扩展机制实现，不污染核心字段。

## 3. 整体分层架构与职责

平台从上至下共分为 **6 个逻辑独立层级**，每层职责单一、边界清晰，支持独立开发、测试、部署、扩缩容。

```
                         ┌──────────────────────────────────────┐
                         │           上游异构数据源层            │
                         │  中国市场 · 印度 · 韩国 · 香港 · GMI  │
                         └────────────────┬─────────────────────┘
                                          ▼
                         ┌──────────────────────────────────────┐
                         │         适配器标准化层 (Adaptor)       │
                         │  解析 → 清洗 → 推导 → 封装 → Proto    │
                         └────────────────┬─────────────────────┘
                                          ▼
                         ┌──────────────────────────────────────┐
                         │        Kafka 统一消息总线层           │
                         │  标准 Topic · 重放 · 回填 · 断点续传   │
                         └────────────────┬─────────────────────┘
                                          ▼
                         ┌──────────────────────────────────────┐
                         │          计算模块层（业务核心）         │
                         │   市场专属模块 + 全局跨市场汇总模块     │
                         └──────────┬───────────────┬───────────┘
                                    ▼                               ▼
       ┌─────────────────────────────────┐   ┌──────────────────────────────────┐
       │        下游结果输出层（流式）       │   │     查询服务层 (Query Service)     │
       │  OSM · Aviator · FOX · 风控 · 清算│   │   CQRS 读写分离 · REST/gRPC 接口    │
       │  直接消费结果 Kafka Topic          │   │  物化视图 · 只读快照 · 最终一致性   │
       └─────────────────────────────────┘   └──────────────────────────────────┘
```

### 3.1 上游异构数据源层

汇聚所有业务市场原始数据，不同市场使用不同协议、结构与语义：

| 市场 | 数据源 | 协议 |
|------|--------|------|
| 中国市场 | Genie 实时 Kafka 流 + GMI 数据库日初快照 (SOD) | Kafka / DB |
| 印度市场 | Precision HTTP 接口数据 | HTTP |
| 韩国市场 | Koscom 交易所 HTTP 数据 | HTTP |
| 香港市场 | HKEX 交易所私有 HTTP 数据流 | HTTP |
| GMI 数据库 | 每日开盘持仓、保证金静态快照数据 | DB |

### 3.2 适配器标准化层 (Adaptor)

每个市场 / 数据源对应独立的适配器服务，负责原始数据解析、清洗、标准化、格式转换。

#### 3.2.1 现有适配器清单

- **Harbour China Adaptor** — 中国市场适配器
- **Harbour India Adaptor** — 印度市场适配器
- **Harbour Korea Adaptor** — 韩国市场适配器
- **Harbour HKEX Adaptor** — 港交所适配器
- **GMI DB Adaptor** — 专门用于日初 SOD 快照初始化

#### 3.2.2 核心职责

解析上游原始数据，完成清洗、字段映射、语义封装，转换为平台标准 Protobuf 消息，最终推送至 Kafka 消息总线。

#### 3.2.3 允许执行的操作（白名单）

> 仅允许数据预处理、格式转换、语义封装，禁止任何业务计算。

- 字段映射、数据类型转换、时间戳统一格式化；
- 基于数据完整性规则过滤脏数据（空值、非法枚举、越界数据）；
- 从原始成交流水推导统一模型必需基础字段（如根据成交方向 + 数量计算持仓变动）；
- 将市场私有业务语义打包为标准扩展消息 (`extensions`)。

#### 3.2.4 严格禁止的操作

- 保证金、盈亏、风险指标等业务规则计算；
- 跨多条记录、跨账户的持仓聚合、对冲合并、头寸归并；
- 子账户层级关系计算与推导。

> **约束落地**：通过 Code Review + 自动化检查双保障，防止职责越界。

### 3.3 Kafka 统一消息总线层

平台数据核心枢纽，承担数据传输、缓冲、回放、回填、故障恢复能力，全链路仅传输标准 Protobuf 数据。

#### 3.3.1 标准 Topic 划分（按数据类型 + 时间维度）

| Topic | 说明 |
|-------|------|
| `harbour.unified.position.sod` | 日初快照持仓数据 |
| `harbour.unified.position.intraday` | 日内增量实时持仓数据 |
| `harbour.unified.margin.sod` | 日初快照保证金数据 |
| `harbour.unified.margin.intraday` | 日内增量实时保证金数据 |

#### 3.3.2 基础能力

支持数据重放、历史回填、故障断点续传；所有 Topic 数据结构全局统一，**不区分市场**。

#### 3.3.3 SOD 与 Intraday 分 Topic 设计深度解析

> 为什么 SOD 和 Intraday 必须是两个 Topic？本节从 6 个维度论证分开设计的必要性。

**两种数据的本质差异**：

| 属性 | SOD (Start of Day) | Intraday (日内增量) |
|------|-------------------|---------------------|
| **来源** | GMI DB 快照 | 实时适配器（Genie Kafka 流等） |
| **频率** | 每天 1 次（开盘前） | 交易时段持续产生 |
| **体量** | 全量快照（所有账户一次性） | 单笔成交粒度 |
| **语义** | 完整持仓基线 | 增量变动 |
| **延迟要求** | 分钟级可容忍 | 毫秒级 |
| **可靠性** | 错了可以等修正版重发 | 丢了就是永久丢失 |
| **恢复** | 重放今天的 SOD 即可 | 需要 SOD + 重放当天所有 Intraday |

---

**维度一：消费语义完全不同**

SOD 是"建基线"，Intraday 是"增量累加"。如果混在一个 Topic，消费者每条消息都要做类型判断：

```
混合 Topic（❌ 反模式）：
  [SOD  account=A01 qty=100]  ← 基线
  [INT  account=A01 delta=+2] ← 增量
  [INT  account=A02 delta=-5]
  [SOD  account=A02 qty=50]   ← A02 基线出现在增量之后？乱序！
  [INT  account=A01 delta=-1]
  [SOD  account=A01 qty=120]  ← 是修正版 SOD 还是重复消息？
```

消费者被迫维护复杂状态机：这条是 SOD 还是 Intraday？该账户的 SOD 到了吗？这是修正版吗？所有逻辑混在一起。

**分开后**：消费者订阅两个 Topic，SOD Consumer 只关心建基线，Intraday Consumer 只关心增量累加，职责清晰。

---

**维度二：吞吐量特征冲突**

```
                        开盘前 5 分钟          交易时段（4 小时）
                        ──────────────        ──────────────────
SOD Topic               全量 burst            几乎无流量
                        (数千账户 × 几十条持仓)
Intraday Topic          无流量                持续均匀流
                        (还没开盘)             (每秒数百条)
```

如果混在一起，Kafka 分区设计陷入两难：分区数按 SOD 峰值设计则浪费资源，按 Intraday 稳态设计则开盘时积压。分开后可独立调优。

---

**维度三：保留策略不同**

| 策略 | SOD | Intraday |
|------|-----|----------|
| **保留期** | 30 天（合规审计需要日初快照） | 7 天（可从 SOD + 后续 Intraday 重建） |
| **压缩策略** | `cleanup.policy=compact`（同 key 只保留最新版本） | `cleanup.policy=delete`（每条增量都有意义，禁止压缩） |
| **删除策略** | 按天整批删除 | 按时间窗口滚动 |

Kafka Topic 的 `retention.ms` 和 `cleanup.policy` 是 Topic 级别的，混在一起只能取两者交集 → 要么 SOD 保留不够、要么 Intraday 浪费存储。

---

**维度四：故障恢复路径完全不同**

```
场景：计算模块挂了 30 分钟，需要恢复

如果分开：
  ① SOD Consumer：从 Changelog 恢复基线状态（秒级）
  ② Intraday Consumer：从上次 committed offset 继续消费 30 分钟的数据（分钟级）
  ③ 合并：基线 + 增量 → 正常状态
  两条链路独立，互不干扰

如果混在一起：
  必须从唯一 Topic 的某个 offset 开始重放
  → 无法跳过 SOD 消息
  → 无法区分哪些是增量、哪些是基线
  → 恢复逻辑极其复杂
```

这与 8.3 节 Changelog 状态恢复机制配合：分开的 Topic 才能独立做 Changelog。

---

**维度五：分区键与分区数策略不同**

| 属性 | SOD | Intraday |
|------|-----|----------|
| **分区键** | `account_id` | `account_id` |
| **分区数** | 8（数据量小，一次性） | 32（按交易时段吞吐量设计） |
| **消息 Key** | `account_id:business_date` | `execution_id` |

虽然都用 `account_id` 分区以保证 co-partition（同一账户的 SOD 和 Intraday 进入同一消费者实例），但分区数不同是有道理的 — SOD 数据量小可以少分区减少开销，Intraday 需要多分区支撑并发消费。分开才能独立调整分区数。

---

**维度六：消息版本与去重语义不同**

| 语义 | SOD | Intraday |
|------|-----|----------|
| **版本控制** | `sod_version` 递增（GMI 修正重发） | 无版本概念 |
| **重复处理** | 同一 `account_id + business_date` 的新版 SOD **覆盖**旧版 | `execution_id` 全局唯一，重复消息**跳过** |
| **消费者行为** | 收到新版 SOD → 回滚当日增量 → 用新版 SOD 重建基线 | 查询去重集合 → 命中则丢弃 |

两种完全不同的去重/版本语义，混在一起消费者要同时维护两套逻辑。

---

**Topic 规划总表**：

```
harbour.unified.position.sod
  ├── 分区键：account_id
  ├── 分区数：8（可调）
  ├── 保留：30 天
  ├── 压缩：cleanup.policy=compact（同 key 保留最新版本）
  ├── 语义：日初基线快照
  └── 版本：sod_version 递增

harbour.unified.position.intraday
  ├── 分区键：account_id
  ├── 分区数：32（可调，按吞吐量）
  ├── 保留：7 天
  ├── 压缩：cleanup.policy=delete（不压缩，每条有意义）
  ├── 语义：日内增量变动
  └── 去重：execution_id

harbour.unified.margin.sod        ← 同上（position → margin）
harbour.unified.margin.intraday   ← 同上（position → margin）
```

---

**唯一不需要"分"的场景**：如果某个市场只有 Intraday 没有 SOD（该市场不提供日初快照，全靠实时流累积），则其 Intraday Topic 设计不变，只是没有对应的 SOD Topic。不存在需要"合成一个 Topic"的情况。

**结论**：SOD 和 Intraday 必须是两个 Topic。这是以数据语义为边界的设计决策，不是拍脑袋。

| 维度 | 分开 | 合在一起 |
|------|:---:|:---:|
| 消费语义清晰度 | ✅ 两个 Consumer 各司其职 | ❌ 单一 Consumer 被迫做类型判断 |
| 吞吐量调优 | ✅ SOD/Intraday 独立扩分区 | ❌ 峰值和稳态无法兼顾 |
| 保留策略 | ✅ SOD 30天压缩 / Intraday 7天不压缩 | ❌ 只能取交集 |
| 故障恢复 | ✅ 两条链路独立恢复 | ❌ 恢复逻辑极其复杂 |
| 分区策略 | ✅ 独立调整分区数与分区键语义 | ❌ 互相掣肘 |
| 版本/去重语义 | ✅ 各自独立管理 | ❌ 两套逻辑混在一起 |
| 运维复杂度 | ✅ 职责分离，问题定位快 | ❌ 一条 Topic 承载两种语义 |

### 3.4 计算模块层（业务核心层）

> 计算模块的完整 Class 架构、状态机实现、Kafka Streams 拓扑、故障恢复等详细设计见独立文档：
> - Position 计算模块：**`calculation-module-architecture.md`**
> - Margin 计算模块（多源聚合 + 人工超控 + 参考数据缓存）：**`margin-module-architecture.md`**

所有计算模块仅订阅 Kafka 统一流，与上游数据源完全解耦；完成业务计算后，将结果再次封装为 Protobuf 消息输出至结果类 Kafka Topic。

#### 3.4.1 模块分类

| 类型 | 模块 | 部署模式 |
|------|------|---------|
| 市场专属计算模块 | 中国持仓/保证金模块、印度模块、韩国模块、港交所模块 | 各市场独立部署，按市场吞吐量独立扩容 |
| 全局跨市场汇总模块 | 全市场持仓、保证金统一聚合汇总模块 | Consumer Group 多实例，按 `account_id` 分区水平扩展 |

#### 3.4.2 全局汇总分片策略

各市场计算模块输出的结果 Topic 统一使用 `account_id` 作为 Kafka 分区键，保证同一账户的跨市场结果消息全部落入同一分区。全局汇总模块以 Consumer Group 多实例模式消费，每个实例负责若干分区，在实例内部完成该账户粒度的跨市场聚合。

```
市场专属模块 A ──→ 结果 Topic (partition=hash(account_id))
市场专属模块 B ──→ 结果 Topic (partition=hash(account_id))
                              │
            ┌─────────────────┼─────────────────┐
            ▼                 ▼                 ▼
    汇总实例 1          汇总实例 2          汇总实例 3
    (分区 0-3)         (分区 4-7)         (分区 8-11)
```

- **按账户有序**：同一 `account_id` 始终由同一分区同一实例处理，保证聚合的严格顺序性；
- **水平扩展**：新增实例时增加分区数并 Rebalance，吞吐线性增长；
- **故障隔离**：单实例故障仅影响其负责的分区账户，其余分区不受影响；
- **跨账户全局指标**（如全市场总持仓）：由 Query Service 的物化视图做跨账户 `SUM` 聚合，或增加一层轻量级全局聚合层（消费每个分区的聚合结果做二次汇总），不在流处理层实现。

全局结果 Topic 同样按 `account_id` 分区，下游 Query Service 按相同分区键消费，写入各层缓存。

#### 3.4.3 核心计算能力

- GMI + 多级子账户持仓、保证金聚合计算；
- 日初快照 (SOD) 与日内增量数据合并计算；
- 期货、期权全品类持仓计算，期权冻结持仓、保证金优惠逻辑；
- 各市场专属保证金规则计算（中国阶梯保证金、印度 SPAN、韩国 KRX 规则等）；
- 跨市场统一风险指标统计。

### 3.5 下游结果输出层

计算完成的标准 Protobuf 数据推送至结果类 Kafka Topic，供流式下游系统直接消费：

- **OSM** — 订单结算管理系统
- **Aviator** — 保证金 / 持仓监控 UI
- **FOX** — 持仓查询系统
- 内部风控、清算系统

### 3.6 查询服务层（Query Service / CQRS 读写分离）

为同步查询场景设计，解决纯流式架构无法满足低延迟点查的问题，读写链路物理隔离。详细规则见[第 10 章](#10-查询服务层cqrs-读写分离)。

### 3.7 SOD 与日内增量数据合并时序规范

日初快照 (SOD) 来自 GMI DB Adaptor，日内增量 (Intraday) 来自实时适配器，两者到达时序无保证，计算模块必须处理以下场景。

#### 3.7.1 关键时序场景

| 场景 | 说明 | 处理策略 |
|------|------|---------|
| SOD 先到，Intraday 后到 | 正常路径 | SOD 建立基线 → Intraday 增量叠加 |
| Intraday 先到，SOD 后到 | 开盘后 GMI 快照延迟就绪 | 缓存 Intraday → SOD 到达后回放合并，合并完毕前标记该账户为 `initializing` |
| SOD 与 Intraday 同时到达 | 并发写入 | 基于 `business_date` 分区锁，同一交易日同一账户串行合并 |
| SOD 数据修正重发 | GMI 发布修正版日初快照 | SOD 消息携带 `sod_version`，版本号递增；收到更高版本时回滚该交易日的增量合并，以新版 SOD 重建基线 |

#### 3.7.2 去重策略

同一笔成交可能同时出现在 SOD 和 Intraday 中（例如 GMI 快照时间窗口与实时流重叠）。计算模块使用 **`execution_id`**（见 [5.3.1](#531-核心消息定义永久不变)）进行去重：

- SOD 加载时记录所有 `execution_id` 到当日去重集合；
- Intraday 消息到达时先查去重集合，命中则跳过；
- 去重集合在下一个交易日 SOD 到达时清空重建。

#### 3.7.3 状态标记

每个账户在每个交易日有三种合并状态，通过查询接口暴露：

| 状态 | 含义 | 查询行为 |
|------|------|---------|
| `initializing` | SOD 未就绪，仅累积了 Intraday 增量 | 返回临时结果 + 状态标记，调用方可选择等待 |
| `ready` | SOD 已就绪，增量合并正常 | 返回最终结果 |
| `stale` | 超过阈值时间未收到任何数据 | 返回最后已知结果 + 告警标记 |

## 4. 端到端完整数据流

1. 各市场生成原始业务数据（协议、结构互不相同）；
2. 对应适配器解析原始数据，完成清洗、字段推导、市场语义封装；
3. 适配器将数据转换为统一 Protobuf 消息（核心字段 + 扩展字段）；
4. 标准化消息发布至 Kafka 统一总线对应 Topic；
5. 计算模块订阅 Kafka 流，执行子账户聚合、持仓合并、保证金计算等业务逻辑；
6. 计算结果封装为标准 Protobuf，输出至结果类 Kafka Topic；
7. 下游分为两条消费链路：
   - **流式链路**：结算、风控等系统直接消费结果 Kafka 流；
   - **查询链路**：UI、查询类系统调用查询服务 REST/gRPC 接口获取快照数据。

## 5. 统一数据模型设计（核心 + 扩展机制）

### 5.1 设计背景

多市场存在大量市场特有业务语义：

| 市场 | 特有语义 |
|------|---------|
| 中国 | 今仓 / 昨仓区分、备兑标记（用于手续费、保证金优惠） |
| 印度 | SPAN 参数组、组合价差持仓信息 |
| 通用期权 | 冻结量、备兑冻结量 |

- 若将所有市场字段**平铺至核心模型** → 模型臃肿、语义污染、新增市场必须修改核心结构；
- 若直接**丢弃特有字段** → 计算层无法完成业务逻辑。

### 5.2 扩展机制设计原则

| 原则 | 说明 |
|------|------|
| **核心归核心** | 全市场通用字段放入核心消息，结构永久稳定，不因市场迭代变更 |
| **扩展归扩展** | 市场特有语义独立存放，不侵入核心字段 |
| **隔离消费** | 通用模块仅读取核心字段，市场专属模块按需解析扩展消息 |
| **治理闭环** | 扩展名称、类型统一注册，适配器、CI 强制校验，杜绝非法注入 |

### 5.3 技术方案：`map<string, google.protobuf.Any>` 扩展槽

采用 Protobuf3 + `google.protobuf.Any` 实现类型安全、零侵入扩展。

#### 5.3.1 核心消息定义（永久不变）

```protobuf
// unified_position.proto
syntax = "proto3";
import "google/protobuf/any.proto";
import "harbour/common/enums.proto";

message UnifiedPosition {
    // 全局核心公共字段（所有市场通用）
    string execution_id = 1;        // 全局唯一成交标识，用于去重与审计追踪
    string account_id = 2;
    string sub_account_id = 3;
    string symbol = 4;
    string mic = 5;                 // ISO 市场编码
    string venue = 6;               // 同一 MIC 下可能有多个交易场所
    PositionDirection direction = 7;
    string quantity = 8;            // 金额/数量以字符串传输，避免浮点精度丢失
    string avg_price = 9;
    int64 timestamp = 10;           // 物理时间戳（成交发生时刻）
    string business_date = 11;      // 交易日（YYYYMMDD），与 timestamp 解耦，对 SOD 合并至关重要
    PositionType type = 12;         // 品种类型：期货/期权
    string source = 13;             // 数据来源适配器标识，用于问题回溯

    // 唯一统一扩展入口
    map<string, google.protobuf.Any> extensions = 20;
}
```

#### 5.3.2 市场 / 业务独立扩展消息

按市场、业务维度拆分扩展文件，完全解耦：

```protobuf
// china_extensions.proto — 中国市场扩展
syntax = "proto3";
package harbour.ext;

message ChinaPositionExt {
    string today_qty = 1;       // 今仓
    string sod_qty = 2;         // 昨仓
    bool is_covered = 3;        // 备兑标记
}
```

```protobuf
// india_extensions.proto — 印度市场扩展
syntax = "proto3";
package harbour.ext;

message IndiaPositionExt {
    string span_param_group = 1;
    repeated string combined_spreads = 2;
}
```

```protobuf
// option_extensions.proto — 期权通用扩展（跨市场复用）
syntax = "proto3";
package harbour.ext;

message OptionExt {
    string frozen_qty = 1;
    string today_frozen_qty = 2;
    string sod_frozen_qty = 3;
}
```

#### 5.3.3 模块消费规则（代码示例）

**通用聚合模块**：仅使用核心字段，完全不感知扩展

```java
UnifiedPosition pos = ...;
BigDecimal totalQty = new BigDecimal(pos.getQuantity());
// 执行通用子账户合并、全局汇总逻辑
```

**市场专属模块**：按需强类型解析扩展消息

```java
// 中国市场计算模块示例
if (pos.getExtensionsMap().containsKey("china")) {
    Any any = pos.getExtensionsMap().get("china");
    ChinaPositionExt ext = any.unpack(ChinaPositionExt.class);
    // 使用今仓、昨仓、备兑标记计算保证金
}
```

#### 5.3.4 扩展键枚举安全层（编译期防护）

`map<string, Any>` 的 Key 是自由字符串，拼写错误（`"china"` vs `"chian"`）是运行时 bug。在此之上增加枚举注册表，将 Key 映射为整数枚举，适配器与计算模块通过枚举常量引用扩展，拼写错误在编译期即可发现：

```protobuf
// extension_key.proto
syntax = "proto3";
package harbour.ext;

enum ExtensionKey {
    EXT_UNKNOWN = 0;            // 保留，禁止使用
    EXT_CHINA_POSITION = 1;
    EXT_INDIA_POSITION = 2;
    EXT_OPTION = 3;
}
```

适配器写入时使用枚举名作为 map key：

```java
// 编译期安全：ExtensionKey.EXT_CHINA_POSITION.name() → "EXT_CHINA_POSITION"
pos.putExtensions(
    ExtensionKey.EXT_CHINA_POSITION.name(),
    Any.pack(chinaExt)
);
```

计算模块读取时同样使用枚举常量，避免魔法字符串：

```java
if (pos.containsExtensions(ExtensionKey.EXT_CHINA_POSITION.name())) {
    ChinaPositionExt ext = pos.getExtensionsMap()
        .get(ExtensionKey.EXT_CHINA_POSITION.name())
        .unpack(ChinaPositionExt.class);
}
```

> **注意**：枚举 Key 是 `map<string, Any>` 之上的**命名规范层**，不改变 wire format，向后兼容。枚举定义表本身即注册中心的核心数据结构。

### 5.4 方案选型对比

| 方案 | 结论 | 原因 |
|------|:----:|------|
| 平铺 `optional` 字段 | ❌ 淘汰 | 模型膨胀、语义污染、新增市场必须改核心结构 |
| `oneof` 单选分支 | ❌ 淘汰 | 新增市场仍需修改核心消息，稳定性不足 |
| `map<string, Any>` | ✅ 采用 | 核心永久稳定、扩展独立、类型安全、新增市场零侵入 |

### 5.5 扩展机制治理规范

#### 扩展注册中心

维护统一注册表 (YAML)，记录扩展 Key 与 Protobuf 类型映射，所有扩展必须先注册再使用：

```yaml
# extensions_registry.yaml
# string_key: [enum_key, proto_message_type]
extensions:
  china:   [EXT_CHINA_POSITION, harbour.ext.ChinaPositionExt]
  india:   [EXT_INDIA_POSITION, harbour.ext.IndiaPositionExt]
  option:  [EXT_OPTION,        harbour.ext.OptionExt]
```

#### 治理规则

- **适配器强制校验**：禁止使用未注册的 string Key 或枚举 Key、禁止塞入非 Protobuf 裸数据；写入时必须同时使用枚举常量引用（`ExtensionKey.EXT_XXX.name()`），CI 扫描禁止裸字符串调用 `putExtensions("china", ...)`；
- **依赖声明**：每个市场模块必须文档标注依赖的扩展 Key（string + enum）、消息类型；
- **CI 自动化检查**：流水线校验扩展 Key 已注册、枚举引用有效、类型合法，非法代码禁止合并；
- **版本兼容**：扩展消息遵循 Protobuf 向前兼容规则，迭代不影响核心与其他市场；新增扩展 Key 时同步追加 `ExtensionKey` 枚举值与注册表条目。

### 5.6 Schema Registry：消息 Schema 全生命周期治理

YAML 扩展注册表（见 5.5）仅管理扩展 Key ↔ 消息类型的映射关系，**不管理 Schema 二进制兼容性**。在多团队协作下（适配器团队独立迭代扩展消息、计算模块团队独立升级消费端），字段类型变更、字段删除、字段号冲突等兼容性破坏可能在上线后才暴露。

#### 5.6.1 架构

```
适配器 ──→ 序列化时查询 Schema Registry ──→ Kafka (带 Schema ID)
计算模块 ──→ 反序列化时查询 Schema Registry ──→ 校验通过后处理
Query Service ──→ 反序列化时查询 Schema Registry ──→ 校验通过后构建视图
```

#### 5.6.2 技术要求

| 要求 | 说明 |
|------|------|
| Schema Registry 选型 | Confluent Schema Registry（原生支持 Protobuf） |
| 兼容性模式 | `FULL`（同时检查向前和向后兼容），禁止破坏性变更进入生产 |
| 注册范围 | 所有 Kafka Topic 使用的 Protobuf 消息类型：核心消息（`UnifiedPosition`）+ 每种扩展消息（`ChinaPositionExt`、`IndiaPositionExt`、`OptionExt` 等） |
| Subject 命名 | `topic-name-value`（如 `harbour.unified.position.intraday-value`） |
| CI 集成 | 流水线在构建阶段执行 `protoc` 编译 + Schema Registry 兼容性预检（dry-run validate），不兼容则阻断合并 |

#### 5.6.3 与 YAML 扩展注册表的打通

YAML 注册表与 Schema Registry 互补：

| 注册表 | 管理内容 | 检查时机 |
|--------|---------|---------|
| **YAML 扩展注册表** | 扩展 Key → 枚举值 → 消息类型的映射关系 | CI 静态扫描 + 适配器运行时校验 |
| **Schema Registry** | Protobuf 消息二进制的字段兼容性（类型、字段号、删除/新增规则） | 构建时 CI 预检 + 生产者注册/消费者反序列化时实时校验 |

两者的 subject 和版本信息统一存储，形成平台的**统一元数据中心**。YAML 中的 `proto_message_type` 字段引用 Schema Registry 中的对应 subject。

### 5.7 数据模型设计深度解析：核心归核心，扩展归扩展

> 本节对第 5 章的扩展机制做进一步设计哲学拆解，回答「为什么这样设计」「如何保证不把 message 塞太满」「如果不这样做还有什么选择」。

#### 5.7.1 问题本质：字段的交集 vs 并集

多市场字段天然存在巨大差异，以下为四个市场 + 期权通用的字段矩阵：

```
中国市场: execution_id, account_id, symbol, ..., today_qty(今仓), sod_qty(昨仓), is_covered(备兑)
印度市场: execution_id, account_id, symbol, ..., span_param_group, combined_spreads
韩国市场: execution_id, account_id, symbol, ..., (无特殊字段)
香港市场: execution_id, account_id, symbol, ..., (无特殊字段)
期权通用: execution_id, account_id, symbol, ..., frozen_qty, today_frozen_qty, sod_frozen_qty
```

- **所有市场的交集** → 只有 `execution_id`, `account_id`, `symbol` 等约 13 个字段
- **所有市场的并集** → 30+ 字段，其中 60% 对大多数市场是 `null`
- **平铺并集方案** → 核心 message 臃肿、语义污染、新增市场必须修改核心 `.proto` 导致所有下游重新编译部署

Harbour 的方案：**核心 message 只存交集，差集塞进 `map<string, google.protobuf.Any>` 扩展槽**。

#### 5.7.2 两层结构拆解

**第一层：核心字段 — 永不变更的公共属性**

核心字段的入选标准（三条必须同时满足）：

| 标准 | 含义 |
|------|------|
| **全市场共有** | 所有市场都有这个语义（即使源字段名不同，适配器做了映射） |
| **通用模块依赖** | 跨市场汇总等通用计算模块需要读取它（如 `account_id`、`symbol`、`quantity`） |
| **永久不废弃** | 是持仓/保证金业务的基本原子，不会因业务演变而被移除 |

```protobuf
message UnifiedPosition {
    string execution_id = 1;    // 全市场：唯一成交 ID（去重、审计追踪）
    string account_id = 2;      // 全市场：账户
    string sub_account_id = 3;  // 全市场：子账户
    string symbol = 4;          // 全市场：合约代码
    string mic = 5;             // 全市场：ISO 市场编码
    string venue = 6;           // 全市场：交易场所（同一 MIC 可能有多个）
    PositionDirection direction = 7;  // 全市场：买卖方向
    string quantity = 8;        // 全市场：数量（字符串避免浮点精度丢失）
    string avg_price = 9;       // 全市场：均价
    int64 timestamp = 10;       // 全市场：成交时间戳
    string business_date = 11;  // 全市场：交易日 YYYYMMDD
    PositionType type = 12;     // 全市场：期货/期权
    string source = 13;         // 全市场：来源适配器标识

    map<string, google.protobuf.Any> extensions = 20;  // ← 唯一扩展入口
}
```

> 该 message 一旦稳定，便不再修改。新增市场接入时，核心字段一个不动。

**第二层：扩展槽 — 每个市场独立维护的私有字段**

每个扩展是**独立的 `.proto` 文件**，有自己的 `package`，编译后是完全解耦的类：

```protobuf
// china_extensions.proto — 只属于中国市场
package harbour.ext;
message ChinaPositionExt {
    string today_qty = 1;       // 今仓
    string sod_qty = 2;         // 昨仓
    bool is_covered = 3;        // 备兑标记
}

// india_extensions.proto — 只属于印度市场
package harbour.ext;
message IndiaPositionExt {
    string span_param_group = 1;
    repeated string combined_spreads = 2;
}

// option_extensions.proto — 跨市场复用的期权扩展
package harbour.ext;
message OptionExt {
    string frozen_qty = 1;
    string today_frozen_qty = 2;
    string sod_frozen_qty = 3;
}
```

关键设计点：

- 每个扩展文件独立编译，新增韩国特有字段只需新建 `KoreaPositionExt`，不动 `ChinaPositionExt`，更不动 `UnifiedPosition`
- `OptionExt` 是「跨市场复用」的扩展（中国期权、印度期权都需要），但不放入核心字段 — 因为期货市场不需要

#### 5.7.3 运行时消费：隔离原则（核心设计精髓）

不同模块**读不同的层**，这是保持扩展性同时不污染核心的关键：

```java
// ─── 通用跨市场汇总模块 ───
// 只读核心字段，完全不感知扩展，不 import 任何扩展类
UnifiedPosition pos = consumer.next();
BigDecimal qty = new BigDecimal(pos.getQuantity());
aggregator.add(pos.getAccountId(), pos.getSymbol(), qty);
// ↑ 这段代码对 4 个市场完全相同，一行不用改

// ─── 中国市场专属模块 ───
// 读到核心字段 + 按需解析中国扩展
if (pos.containsExtensions(ExtensionKey.EXT_CHINA_POSITION.name())) {
    ChinaPositionExt ext = pos.getExtensionsMap()
        .get(ExtensionKey.EXT_CHINA_POSITION.name())
        .unpack(ChinaPositionExt.class);
    BigDecimal todayQty = new BigDecimal(ext.getTodayQty());
    BigDecimal sodQty = new BigDecimal(ext.getSodQty());
    // 基于今仓/昨仓差异计算阶梯保证金...
}
```

```
消费隔离示意：

  通用模块                      中国市场模块
  ┌──────────────────┐       ┌──────────────────────────┐
  │ 只读核心字段       │       │ 读核心字段                 │
  │ 不 import 任何扩展 │       │ + 解析 ChinaPositionExt   │
  │ 永远不因新市场改动  │       │ + 执行阶梯保证金逻辑       │
  └──────────────────┘       └──────────────────────────┘
```

#### 5.7.4 四层治理防线：防止扩展槽变成垃圾场

`map<string, Any>` 本身只是自由键值对，如果没有治理，最终会变成充斥拼写错误、未注册类型、废弃字段的垃圾场。Harbour 设计了四层递进防线：

| 防线 | 层级 | 机制 | 拦截内容 |
|------|------|------|---------|
| **① 编译期** | 代码 | `ExtensionKey` 枚举，禁止魔法字符串 | 拼写错误（`"china"` vs `"chian"`）、未定义 Key |
| **② CI 静态扫描** | 流水线 | 扫描 `putExtensions` 调用，对照 YAML 注册表 | 未注册 Key、类型不匹配、裸字符串调用 |
| **③ 构建时** | CI | Schema Registry `FULL` 兼容性预检 | 字段类型变更、字段号冲突、破坏性删除 |
| **④ 运行时** | 适配器 | 启动时加载注册表到内存，写入前检查 | 绕过 CI 的未注册 Key（最后一道防线） |

**防线 ①：ExtensionKey 枚举（编译期安全）**

```java
// ❌ CI 会拦截 — 裸字符串，拼写错误编译期不报错
pos.putExtensions("china", any);

// ✅ 唯一合法写法 — 枚举常量，拼写错误编译期暴露
pos.putExtensions(ExtensionKey.EXT_CHINA_POSITION.name(), any);
```

**防线 ②：YAML 注册表 + CI 静态扫描**

```yaml
# extensions_registry.yaml — 扩展的唯一真实来源
extensions:
  china:   [EXT_CHINA_POSITION, harbour.ext.ChinaPositionExt]
  india:   [EXT_INDIA_POSITION, harbour.ext.IndiaPositionExt]
  option:  [EXT_OPTION,        harbour.ext.OptionExt]
```

CI 扫描规则：
- 所有 `putExtensions(key, any)` 调用的 key 必须在注册表中
- `any.pack()` / `any.unpack()` 的泛型类型必须与注册表声明一致
- 禁止使用 `putExtensions` 的字符串重载（强制走枚举路径）

**防线 ③：Schema Registry FULL 兼容性**

不同于 YAML 注册表管理「Key → 类型」映射，Schema Registry 管理「Protobuf 二进制兼容性」：

- `FULL` 模式：同时检查向前（新生产者 → 旧消费者）和向后（旧生产者 → 新消费者）兼容
- 只允许：新增 optional 字段、删除 optional 字段（谨慎）、新增枚举值
- 禁止：修改字段类型、修改字段号、删除 required 字段
- CI 预检：`protoc` 编译后 dry-run 验证，不兼容则阻断合并

**防线 ④：适配器运行时启动检查**

```java
// 适配器启动时加载注册表
ExtensionRegistry registry = ExtensionRegistry.loadFromYaml("extensions_registry.yaml");

// 每次写入扩展前检查
public void putExtension(UnifiedPosition.Builder builder, ExtensionKey key, Any value) {
    if (!registry.isRegistered(key)) {
        throw new IllegalExtensionException("未注册的扩展 Key: " + key);
    }
    if (!registry.matchesType(key, value)) {
        throw new IllegalExtensionException("扩展类型不匹配: expected " 
            + registry.getType(key) + ", got " + value.getTypeUrl());
    }
    builder.putExtensions(key.name(), value);
}
```

#### 5.7.5 方案选型完整对比

| 方案 | 结论 | 核心缺陷 |
|------|:----:|---------|
| **平铺 `optional` 字段** | ❌ | 核心 message 膨胀到 30+ 字段；新增市场必须修改核心 `.proto`，所有下游重新编译部署；字段语义归属模糊 |
| **`oneof` 分支** | ❌ | 每条消息只能有一个市场的数据，跨市场汇总模块无法同时处理多市场；新增市场仍需修改核心 message |
| **每个市场独立 message** | ❌ | 失去统一性，Kafka Topic 里存多种类型，通用模块无法消费，需要类型判断分支 |
| **JSON 自由字段** | ❌ | 丢掉类型安全，字段拼写错误运行时才暴露；序列化体积大、性能差；无 Schema Registry 兼容性校验 |
| **`map<string, Any>` + 治理** | ✅ | 核心永久 13 个字段、扩展独立文件、类型安全、新增市场零侵入核心、四层治理防退化 |

#### 5.7.6 设计决策总结

```
                     ┌──────────────┐
                     │  新市场接入   │
                     └──────┬───────┘
                            │
              ┌─────────────┼─────────────┐
              │             │             │
              ▼             ▼             ▼
       核心字段是否    扩展字段是否    保证金规则
       需要新增？      需要新增？      是否不同？
              │             │             │
              ▼             ▼             ▼
         ❌ 不需要      ✅ 新建扩展     ✅ 新建插件
        (交集已覆盖)   .proto 文件    注册到 ProcessorReg
                       + 注册表一行
```

**核心优势**：

1. **核心 message 永远 13 个字段** — 不管接多少个市场，`UnifiedPosition` 都不变
2. **新增市场零侵入核心** — 只加一个扩展 `.proto` 文件 + 注册表一行 YAML
3. **消费端按需解析** — 通用模块不碰扩展，专属模块只解析自己关心的
4. **拼写错误编译期暴露** — 枚举 Key 替代魔法字符串
5. **CI 堵死非法注入** — 注册表 + 静态扫描 + Schema Registry，不注册就合不进去
6. **扩展独立演进** — 中国市场扩展升级字段不影响印度、韩国、香港的消费者

> 这一设计模式源自 Protobuf 生态的标准实践 — gRPC 的 `grpc-gateway` 转码、Google Cloud API 的 `google.api.HttpBody`、Istio 的 `EnvoyFilter` 均使用相同的 `map<string, Any>` 扩展机制处理多版本/多场景差异化数据。

## 6. 新市场接入与代码复用规则

平台最大化复用通用能力，区分可复用部分与必须新增部分，避免"零修改"误导。

### 6.1 复用与新增明细

| 能力分类 | 是否可复用 | 说明 |
|---------|:---------:|------|
| 公共计算框架 | ✅ 完全复用 | 子账户聚合、SOD 初始化、日内增量合并、通用风控框架 |
| 跨市场全局汇总模块 | ✅ 完全复用 | 全市场持仓/保证金聚合逻辑，无需改动 |
| 适配器标准化流程 | ✅ 复用模式 | 统一流程：解析 → 清洗 → 扩展封装 → 输出 Proto |
| Kafka 总线 & 下游接口 | ✅ 完全复用 | Topic 结构、数据格式全局统一，下游无需适配 |
| 市场专属保证金规则 | ❌ 必须新增 | 各市场规则不同（SPAN/阶梯/KRX），需独立开发插件/模块 |

### 6.2 新市场接入工作量总结

新增一个市场，仅需完成 **3 项工作**：

1. 开发对应市场适配器；

3. 增加扩展消息解析逻辑（如有市场特有语义）。

> 对比传统架构（每个市场独立开发全套适配+计算+接口），新市场接入仅需完成适配器 + 保证金插件 + 扩展解析，重复开发模块量（公共计算框架、Kafka 总线、下游接口、全局汇总）全部复用，开发工作量大幅降低（具体比例取决于新市场的保证金规则复杂度，此处为定性估计）。

## 7. 过渡期双模型治理方案

项目过渡期同时存在两套数据模型，用于快速交付与长期架构落地，严格管控避免技术债务。

### 7.1 两套模型定位

| 模型 | 类型 | 用途 | 生命周期 |
|------|------|------|---------|
| **临时模型** | RESTful Java POJO | 对外提供 REST 接口，支撑 OSM 从旧 FTP 链路迁移 | 过渡设施，计划下线 |
| **最终模型** | Protobuf (Octane) | 平台内部 Kafka 流、适配器、计算模块标准模型 | 长期唯一标准 |

临时模型包含类：`PositionRecord`、`FuturesPositionRecord`、`OptionPositionRecord`、`MarginRecord`、`ExtendedAccountMap`、`ExtendedAccount`。

### 7.2 强制治理规则

1. POJO 模型标记 `@Deprecated`，明确下线时间，仅用于过渡；
2. **POJO 由 Proto 定义自动生成，禁止手动维护**：使用 `protoc` 插件（或自定义代码生成器）从 `.proto` 文件自动生成 REST 接口使用的 DTO 类。所有字段变更必须先在 Protobuf 定义，再运行代码生成器产出新 POJO。CI 流水线校验 POJO 文件与生成结果一致，手工修改的 POJO 禁止合并；
3. 若对外暴露 Proto 生成的原始类不合适（如担心下游耦合 Proto 内部结构），可加一层薄映射层（MapStruct / 手动 Mapper），但映射层的源类型仍然来自 Proto 生成类，不维护独立 POJO；
4. 收敛方案：逐渐将 REST 接口底层直接对接 Proto 生成的类，最终下线所有中间映射层，以 Protobuf 为唯一数据源。

### 7.3 潜在风险规避

- ❌ 双模型不同步、字段漂移；
- ❌ 业务逻辑长期依赖临时 POJO，架构无法统一；
- ❌ 后期切换成本抬高。

> 通过上述规则全链路管控，彻底规避风险。

## 8. 流处理语义与幂等性规范

### 8.1 Exactly-Once 语义边界（明确范围，杜绝预期偏差）

| 层级 | 语义 | 说明 |
|------|------|------|
| **流处理层** | Exactly-Once | 基于 Kafka Streams，消费 → 计算 → 生产闭环内实现 Offset 原子提交、事务输出，保证流内部无重复处理、无状态丢失 |
| **业务幂等** | 要求携带幂等键 | 所有输出消息必须携带 `idempotency_key`，正式定义如下：

```
idempotency_key = source + ":" + account_id + ":" + business_date + ":" + sequence
```

各字段含义：
- `source`：数据来源适配器标识（见核心消息 5.3.1），防止不同适配器为同一账户生成相同 `sequence` 导致误去重；
- `account_id`：账户标识；
- `business_date`：交易日（YYYYMMDD）；
- `sequence`：适配器层生成的单调递增序号，每个 `(source, account_id, business_date)` 组合内从 1 开始自增，由适配器维护。

下游统一以此完整键实现去重。计算模块出向消息继承适配器的幂等键，新产生的计算结果消息用 `计算模块标识 + account_id + business_date + 自增序号` 构造新幂等键 |
| **外部交互** | At-Least-Once | 对接数据库、第三方服务时仅保证至少一次，依赖数据库唯一约束 / Upsert 语义防重，不承诺端到端精确一次 |
| **数据回填与重放** | 最终一致 | 故障恢复、历史回填会重放消息，短时数据会被覆盖，最终状态与实时计算保持一致 |

### 8.2 落地要求

- 计算模块增加消息幂等校验逻辑；
- 运维文档补充重放、断点恢复操作流程。

### 8.3 状态恢复：本地状态 + Changelog Topic

计算模块（特别是 SOD/Intraday 合并逻辑）维护大量有状态信息——SOD 基线数据、`execution_id` 去重集合、账户合并状态。纯靠从 Kafka 开头重放全量历史来重建状态，在 7 天保留期内最多需要重放 7 天数据，导致故障恢复耗时远超可接受范围（分钟级积压 → 数十分钟重建）。

#### 8.3.1 恢复策略

| 方案 | 适用场景 | 恢复耗时 |
|------|---------|:--------:|
| **Kafka Streams 内置 Changelog** | 使用 Kafka Streams 的计算模块 | 秒级（从 changelog topic 恢复状态存储） |
| **手动 Changelog Topic** | 自行管理状态（如 RocksDB）的计算模块 | 秒级（从 changelog 恢复 + 重放尾部少量消息） |
| **纯 Kafka 回放** | 无 Changelog 的降级路径 | 分钟~小时级（取决于历史数据量） |

#### 8.3.2 手动 Changelog 规范（适用于非 Kafka Streams 方案）

计算模块将每次状态变更写入专用的 compacted changelog topic：

```
harbour.state.changelog.<module_name>
```

- **Compacted topic**：Key 为 `account_id + business_date`，Value 为状态的 Protobuf 序列化快照；
- **写入时机**：每次完成一批消息处理且状态变更后，异步批量写入 Changelog；
- **恢复流程**：重启时优先从 Changelog 消费全量快照重建状态 → 再从原始 Topic 的最近 committed offset 开始消费，补齐 Changelog 写入后到达的尾部消息；
- **Kafka Streams 方案**：配置 `num.standby.replicas ≥ 1`，Standby 实例自动维护热备状态，主实例故障时秒级切换。

#### 8.3.3 运维收益

- 降低对 Kafka 历史保留时间的硬依赖（不再需要保留 7 天全量数据仅用于状态恢复）；
- 故障恢复时间从"重放数十分钟"降为"Changelog 加载 + 尾部补齐 ≈ 数秒"；
- Standby Replica 实现热备，消除单点故障时的状态重建等待。

### 8.4 死信队列 (DLQ) 规范

所有组件在处理单条消息失败且重试耗尽后，必须将失败消息路由至死信队列，禁止静默丢弃。

#### 8.4.1 Topic 命名

```
harbour.dlq.<source_component>
```

| 组件 | DLQ Topic |
|------|----------|
| 中国适配器 | `harbour.dlq.china-adaptor` |
| 印度适配器 | `harbour.dlq.india-adaptor` |
| 中国计算模块 | `harbour.dlq.china-calc` |
| 全局汇总模块 | `harbour.dlq.global-agg` |
| Query Service | `harbour.dlq.query-service` |

#### 8.4.2 消息格式

每条 DLQ 消息封装原始消息 + 失败元数据：

```protobuf
message DLQMessage {
    bytes original_message = 1;       // 原始 Protobuf 序列化消息
    string error_type = 2;            // 异常类名，如 "RefDataException"
    string error_message = 3;         // 异常详情
    string stack_trace = 4;           // 堆栈（截断至前 2KB）
    int32 retry_count = 5;            // 已重试次数
    int64 failure_timestamp = 6;      // 失败时间戳
    string source_component = 7;      // 来源组件
    string trace_id = 8;              // 原始 trace_id，便于关联排查
}
```

#### 8.4.3 重试策略

| 失败类型 | 重试策略 | 说明 |
|---------|---------|------|
| 临时故障（网络超时、RefData 不可用） | 指数退避：100ms → 200ms → 400ms → 800ms → 1.6s（最多 5 次） | 重试耗尽入 DLQ |
| 永久故障（消息格式非法、Schema 不匹配） | 不重试，直接入 DLQ | 需人工介入修复 |
| 业务异常（保证金计算溢出等） | 不重试，直接入 DLQ | 需排查业务逻辑或数据源 |

#### 8.4.4 DLQ 运维

- **告警**：任意 DLQ Topic 消息积压 > 0 触发告警（每条入 DLQ 都意味数据丢失风险）；
- **重放控制台**：提供运维 API `POST /admin/dlq/{topic}/replay`，支持按时间范围或指定 offset 筛选消息，修复后手动回注到正常 Topic；
- **保留期**：DLQ 消息保留 30 天，超期自动清理；
- **监控面板**：Grafana Dashboard 展示各 DLQ Topic 积压量、增长率、top 错误类型。

## 9. 适配器层边界与操作规范

重新明确"数据预处理"与"业务计算"的边界，解决原规则过于绝对、工程落地困难的问题。

核心区分原则：**作用域边界**——单条消息内的推导属于预处理，跨消息 / 跨账户的操作属于业务计算。

### 允许范围（单消息作用域内）

- 字段转换、数据类型映射、时间戳统一格式化；
- 基于数据完整性规则过滤脏数据（空值、非法枚举、越界数据）；
- 从**单条**原始成交流水推导该条消息对应的统一模型基础字段（如根据成交方向 × 数量计算本条持仓变动方向与数量，赋值到 `quantity` 和 `direction` 字段）；
- 将市场私有业务语义打包为标准扩展消息。

### 禁止范围（跨消息 / 跨账户作用域）

- 保证金、盈亏、风险指标等业务规则计算；
- 跨多条记录、跨账户的持仓聚合、对冲合并、头寸归并（多条消息的 `quantity` 求和即为聚合）；
- 子账户层级关系计算与推导；
- 基于历史状态或外部参考数据的任何计算。

### 落地保障

- 代码审查清单 + 自动化规则校验

## 10. 查询服务层（CQRS 读写分离）

### 10.1 建设背景

纯流式架构擅长异步事件处理，无法满足 UI、查询系统、实时风控低延迟同步点查需求；若由各下游自行消费流并构建视图，会造成数据不一致、重复开发。

### 10.2 架构链路

```
计算模块 → Kafka 结果 Topic → Query Service (物化视图) → REST/gRPC 接口
```

### 10.3 核心能力

- 消费结果流，基于分层存储构建以 `account_id` 为索引的账户快照：

| 存储层 | 技术选型 | 用途 | 数据量级 |
|--------|---------|------|:--------:|
| **热数据** | Caffeine 本地缓存 | 当日活跃账户（最近 5 分钟有更新），延迟 < 1ms | ~5,000 账户 |
| **温数据** | Redis Cluster | 近 7 天内有更新的账户，延迟 < 5ms，多实例共享 | ~50,000 账户 |
| **冷数据** | RocksDB 本地持久化 | 全量历史快照，重启无需重放 Kafka 重建全量数据 | 无上限 |

查询流程：先查 Caffeine → 未命中查 Redis → 仍未命中查 RocksDB 并回填热/温层。所有写入同时更新三层（Redis 与 RocksDB 同步写，Caffeine 异步刷新）。
- 提供标准查询接口：`GetPosition`、`GetMargin`；
- 响应头携带 `X-Data-Last-Update-Timestamp` 以及消息对应的 Kafka `offset` 和 `transaction_id`，标识数据版本，便于问题回溯；
- **Kafka Consumer 隔离级别**：必须配置 `isolation.level=read_committed`，确保只消费计算模块事务已提交的消息，避免读到未提交事务的中间态数据造成快照脏读；
- 数据保证**最终一致性**，强一致性场景对接结算库专用接口。

### 10.4 价值

- 统一查询入口、消除下游重复开发；
- 读写链路解耦、不影响核心流处理逻辑。

## 11. 架构约束与运维规范

- 所有市场差异必须在适配器层屏蔽，通用计算模块禁止市场硬编码；
- 严格遵守适配器操作白名单，严禁越界实现业务计算；
- 双模型过渡期强制同步校验，新需求优先基于 Protobuf 模型；
- 流语义边界对外明确，下游必须实现业务幂等；
- 扩展消息严格走注册中心，禁止私自新增扩展 Key；
- 查询服务仅维护只读快照，不修改任何业务数据。

| 高可用 | 支持数据回放、回填、故障恢复，满足金融系统高可用、可审计要求 |
| 长期可维护 | 分层解耦 + 扩展机制，长期架构稳定、可维护、可演进 |

## 13. 安全设计

### 13.1 传输安全

| 链路 | 要求 | 说明 |
|------|------|------|
| 适配器 → Kafka | mTLS + SASL/SCRAM | 防止未授权数据注入，双向证书认证 |
| Kafka → 计算模块 | mTLS | 消费端证书校验，防止未授权消费 |
| 计算模块 → 结果 Topic | mTLS | 生产端证书校验 |
| Query Service ↔ 下游客户端 | TLS 1.3 + API Token | Token 由网关签发，定期轮换 |
| 所有内部组件间 | TLS 1.3 | 杜绝内网明文传输持仓敏感数据 |

### 13.2 认证与鉴权

| 接口类型 | 认证方式 | 鉴权粒度 |
|---------|---------|---------|
| REST 查询接口 | JWT (OAuth2 Client Credentials) | 账户级：仅返回调用方有权查看的 `account_id` |
| gRPC 查询接口 | mTLS + JWT | 同上，同时在拦截器层校验 |
| 管理接口（回放、运维） | mTLS + 独立 Admin Token | 仅允许运维角色 |

### 13.3 审计日志

所有查询操作必须记录不可篡改审计日志：

```
{
  "timestamp": "2026-01-15T10:30:00.123Z",
  "caller": "osm-service",
  "operation": "GetPosition",
  "target_account": "CN_ACC_001",
  "result": "success",
  "trace_id": "a1b2c3d4"
}
```

- 日志写入独立 Kafka Topic (`harbour.audit.log`)，与业务 Topic 物理隔离；
- 保留期 ≥ 90 天，满足金融合规最低要求。

### 13.4 扩展消息注入防护

- 适配器在写入 `extensions` 前，必须通过注册中心校验 Key 与消息类型是否匹配；
- 计算模块在解析扩展消息前，先用 `Any.is(ExpectedType.class)` 校验类型，类型不匹配时记录告警并丢弃该扩展（不丢弃整条核心消息）；
- CI 静态扫描：禁止 `Any.pack()` 调用中传入非注册消息类型。

## 14. 部署与扩容模型

### 14.1 进程 / 容器边界

| 组件 | 部署单元 | 实例数 | 扩缩容策略 |
|------|---------|:-----:|-----------|
| 各市场适配器 | 独立容器/进程 | 1~N | 按上游数据量独立扩容，中国市场可部署 3 实例 |
| Kafka 总线 | 托管集群 | — | 由基础设施团队维护 |
| 通用计算模块 | 独立容器/进程 | 2+ | 基于 CPU 使用率 HPA 自动扩缩 |
| 市场专属计算模块 | 独立容器/进程 | 1~N | 按其所在市场吞吐量独立扩容 |
| 全局汇总模块 | 独立容器/进程 | 2+ | Consumer Group 按 `account_id` 分区消费，水平扩展；故障时对应分区由同组实例接管 |
| Query Service | 独立容器/进程 | 2+ | 无状态，按 QPS HPA 自动扩缩 |

### 14.2 Kafka 分区映射

| Topic | 分区数 | 分区键 | 消费实例上限 |
|-------|:-----:|--------|:-----------:|
| `harbour.unified.position.sod` | 16 | `account_id` hash | 16 |
| `harbour.unified.position.intraday` | 32 | `account_id` hash | 32 |
| `harbour.unified.margin.sod` | 16 | `account_id` hash | 16 |
| `harbour.unified.margin.intraday` | 32 | `account_id` hash | 32 |

> 分区数基于预估峰值吞吐量（10,000 msg/s）设计，按 1 分区 : 300 msg/s 估算。生产环境可根据实测调高。

### 14.3 跨区域部署

| 区域 | 数据驻留要求 | 部署策略 |
|------|:----------:|---------|
| 中国大陆 | 数据不出境 | 独立 Kafka 集群 + 完整 Harbour 实例 |
| 香港 | 可出境至亚太 | 适配器部署在香港，数据经专线发送至亚太中心 Kafka |
| 印度 | 数据不出境 | 独立 Harbour 实例 |
| 韩国 | 可出境至亚太 | 适配器部署在韩国，数据发送至亚太中心 Kafka |

> 跨区域汇总由全局汇总模块的特定实例消费多区域结果 Topic 完成，非实时（T+1 分钟），需要额外网络授权和安全审计。

## 15. 可观测性与监控

### 15.1 链路追踪

#### 标准与实现

采用 **W3C Trace Context** 标准（`traceparent` + `tracestate` header），所有组件统一使用 **OpenTelemetry SDK** 自动注入和提取 trace context。

```
适配器 (OTel SDK) → Kafka Header (traceparent) → 计算模块 (OTel SDK) → 结果 Kafka Header (traceparent) → 下游 / Query Service (OTel SDK)
```

| 组件 | OpenTelemetry 集成方式 | 说明 |
|------|----------------------|------|
| 适配器 | OTel Java Agent 自动埋点 + 手动创建 Span | 自动捕获 Kafka Producer 调用，手动为每条消息创建处理 Span |
| 计算模块 | OTel Java Agent 自动埋点 + 手动创建 Span | 自动捕获 Kafka Consumer/Producer，手动为计算逻辑创建 Span |
| Query Service | OTel Java Agent + gRPC/REST 拦截器 | 自动捕获入站请求和缓存查询 |
| 全局汇总模块 | 同上计算模块 | — |

#### Trace 传递路径

每条消息在适配器入口生成或继承上游的 `trace_id`，在所有组件间透传：

```
适配器入口 (root span)
  └── 字段映射 (child span)
  └── Kafka Produce (child span, inject traceparent to header)
        └── 计算模块消费 (span, extract traceparent from header)
              └── 保证金计算 (child span)
              └── Kafka Produce (child span)
                    └── Query Service 消费 (span)
                          └── 缓存查询 (child span)
```

#### 后端对接

- 通过 OpenTelemetry Collector 将 trace 数据导出至 **Jaeger / Grafana Tempo / Zipkin**，实现端到端调用链可视化，无需手工拼日志定位延迟瓶颈；
- 日志中同时打印 `trace_id` 和 `span_id`，与 ELK / Grafana Loki 集成交叉检索。

### 15.2 核心指标

| 指标类别 | 指标 | 采集方式 | 告警阈值（建议） |
|---------|------|---------|:---------------:|
| 吞吐量 | 适配器输出 msg/s | Prometheus Counter | < 预期值的 50% |
| 吞吐量 | 计算模块处理 msg/s | Prometheus Counter | < 预期值的 50% |
| 延迟 | 适配器 → Kafka 写入延迟 (p99) | Prometheus Histogram | > 50ms |
| 延迟 | 端到端延迟：适配器入口 → 计算结果输出 (p99) | 计算模块消费时对比 `timestamp` | > 500ms |
| 可靠性 | 适配器输入/输出消息数对账 | 每 5 分钟比对 Counter | 差异 > 0.01% |
| 可靠性 | 错误率（解析失败 + 计算异常）/ 总消息数 | Prometheus Counter | > 0.1% |
| 资源 | Query Service 缓存命中率 | Caffeine 内置指标 | < 95% |
| 资源 | CPU / 内存 / 磁盘 | Node Exporter | CPU > 70%, 内存 > 80% |
| 业务 | 账户状态为 `stale` 的占比 | 计算模块上报 | > 5% |
| 业务 | SOD 数据延迟（当前时间 - SOD `business_date` 预期到达时间） | 定时检查 | > 10 分钟 |

### 15.3 健康检查端点

| 端点 | 用途 | 返回 |
|------|------|------|
| `GET /health/live` | Kubernetes Liveness Probe | 进程存活即返回 200 |
| `GET /health/ready` | Kubernetes Readiness Probe | 上游连接正常 + 内部状态就绪时返回 200 |
| `GET /health/deep` | 运维深度检查 | 各依赖（Kafka、DB、Redis）连接状态详情 |

### 15.4 数据对账

- 适配器在每次 Kafka 批量写入成功后，上报 `{source, topic, partition, offset_range, message_count}` 到对账 Topic；
- 计算模块在处理完一批消息后上报消费统计到同一 Topic；
- 独立对账服务每 5 分钟比对上下游计数，差异超过 0.01% 触发告警。

## 16. 测试策略

### 16.1 测试金字塔

| 层级 | 范围 | 覆盖率要求 | 说明 |
|------|------|:--------:|------|

### 16.2 新市场接入测试清单

新增市场时，以下测试必须全部通过：

1. 适配器单元测试：覆盖所有上游原始消息类型 → 标准 Proto 的映射；
2. 扩展消息序列化/反序列化往返测试；
3. 适配器 → Kafka 集成测试：模拟上游数据推送，验证 Topic 写入正确；
4. 市场专属计算模块单元测试：覆盖该市场所有保证金/持仓规则场景；
5. 端到端回放测试：取该市场至少 3 个历史交易日数据完整回放，结果与历史快照对比，偏差 < 0.01%。

## 17. 市场特殊交易日处理

### 17.1 交易日历服务

平台需要独立的市场交易日历服务（或对接已有企业交易日历），为各层提供交易日判定能力：

| 服务能力 | 说明 |
|---------|------|
| `isTradingDay(mic, date)` | 判断指定市场在指定日期是否为交易日 |
| `getPreviousTradingDay(mic, date)` | 获取最近的上一个交易日 |
| `getSODExpectedTime(mic, date)` | 返回该市场 SOD 数据预期到达的时间窗口 |

### 17.2 各层适配规则

| 场景 | 处理规则 |
|------|---------|
| **节假日** | 适配器不产生 Intraday 数据；SOD Adaptor 不产生新的日初快照；计算模块维持前一交易日最终状态，账户标记为 `ready`（非 `stale`） |
| **半天交易日** | 适配器正常推送数据，SOD 快照时间戳为该半日市的开盘时间；计算模块正常合并 |
| **紧急休市** | 适配器检测到交易所状态变更后立即停止推送；已推送的增量数据正常处理；账户状态 5 分钟内不变更则标记为 `stale` |
| **时区对齐** | 各适配器将上游时间戳统一转换为 UTC 后填入 `timestamp` 字段；`business_date` 按各市场的本地交易日历计算（中国 UTC+8，印度 UTC+5:30），全局汇总时按 UTC 时间窗口对齐 |

### 17.3 跨市场汇总的交易日对齐

全局汇总模块消费多市场数据时，各市场的 `business_date` 可能不同（时区差异 + 当地节假日）。汇总逻辑按 **UTC 自然日** 做当日聚合，`business_date` 仅作为来源市场的参考信息透传至下游，不做跨市场对齐。
